## 🎯 CCG 是什么，能帮我们什么

**CCG = Claude + Gemini + Codex**

三种 AI 分工协作，覆盖从"拿到需求"到"代码完成"的全流程。

### 它能帮后端工程师什么？

| 痛点 | 以前 | 现在 |
|------|------|------|
| 拿到模糊需求不知道从哪下手 | 看产品文档，自己琢磨 | AI 帮你解析需求、拆解任务 |
| 技术方案不知道怎么选 | 查文档、问同事、拍脑袋 | AI 给出多方案对比和推荐 |
| 写代码太慢 | 一个个接口手写 | AI 生成 + 你审核 |
| 不知道怎么写测试 | 懒得写 or 随便写 | AI 生成测试用例，你把关 |
| 代码Review容易漏问题 | 靠自己肉眼 review | AI 辅助审查安全问题 |
| 写到一半卡住了 | 卡几小时 | AI 帮你分析卡点，指明方向 |

### 什么场景用 CCG 最有效？

- ✅ 需求不清晰，需要 AI 帮忙分析和拆解
- ✅ 新技术不熟悉，需要 AI 给你讲解和示例
- ✅ 业务逻辑复杂，需要 AI 帮你梳理流程
- ✅ 大量重复代码，需要 AI 批量生成
- ✅ 代码审查，需要 AI 发现潜在问题
- ❌ 简单 CRUD、已有明确规范的不需要

## 🤖 三剑客分工

### 🧠 Claude Opus（指挥官）

**擅长**：理解需求、架构设计、任务拆解、代码审核

```
你：给我分析这个需求，产出技术方案和任务分解
Opus：好的，我先理解需求 → 设计数据库 → 规划 API → 拆解成可执行任务
```

### ✨ Gemini（前端专家）

**擅长**：React/Vue 组件、页面布局、样式优化、仪表盘

```
你：帮我用 React 写一个积分明细页面
Gemini：组件结构、数据流、样式全给你搞定
```

### ⚡ Codex（效率助手）

**擅长**：测试用例、脚本生成、代码补全、重构

```
你：给这个 Service 生成单元测试
Codex：测试类、Mock、断言，一次性给你
```

## ⚡ 5分钟快速上手

### 第一步：安装工具

```bash
# Claude Code（指挥官）
npm install -g @anthropic-ai/claude-code

# Codex（助手）
# 通过 OpenAI API 直接使用，不需要单独安装

# Gemini
# 通过 Google AI Studio 或 API 使用
```

### 第二步：建立工作目录

```
~/projects/
├── backend/      ← Opus + Codex 用这里
├── frontend/     ← Gemini 用这里
└── docs/         ← 需求文档、设计文档
```

> 核心原则：每个 AI 只在自己的目录工作，不跨目录访问，避免上下文混乱。

### 第三步：一次完整流程

```
1. 拿到需求 → 丢给 Opus 分析
2. Opus 产出任务列表 → 你决策
3. Opus 指挥 Gemini 写前端
4. Opus + Codex 写后端和测试
5. Opus 最终审核代码
```

## 💼 日常工作流

### 场景一：拿到一个模糊需求

**以前**：和产品反复确认，耗时几天

**现在**：

```
你 → Opus：
"我需要开发一个'限时折扣订单系统'，需求如下：
[粘贴产品需求文档]
请帮我：
1. 分析业务规则
2. 设计数据库
3. 规划 API 接口
4. 拆解成可执行的任务列表"
```

Opus 会自动产出结构化的需求分析、技术方案、任务分解。

### 场景二：写代码写到一半卡住了

**以前**：卡几小时，问同事不好意思

**现在**：

```
你 → Opus：
"我在实现积分扣减逻辑时遇到问题：
[描述卡点]
请帮我分析：
1. 我的思路哪里可能有问题
2. 有什么解决方案
3. 推荐哪种方案，为什么"
```

Opus 会分析问题，给出多个解决方向，帮你找到出路。

### 场景三：需要写测试但不想自己写

**以前**：测试随便写，覆盖率很低

**现在**：

```
你 → Codex：
"为 UserService 的 addPoints 方法生成测试用例，
覆盖：正常加分、超过上限被拦截、余额不足扣减失败"
```

Codex 生成完整测试，你负责审核逻辑是否正确。

### 场景四：代码 Review

**以前**：肉眼 review，容易漏问题

**现在**：

```
你 → Opus：
"帮我 review 这个 PR，重点检查：
1. 安全性（注入、越权）
2. 并发问题
3. 异常处理是否完整"
```

Opus 逐文件分析，给出具体问题和建议。

### 场景五：不知道该选哪个技术方案

**以前**：拍脑袋决定 or 纠结很久

**现在**：

```
你 → Opus：
"我需要在 MySQL 和 MongoDB 之间选择，
场景：存储用户积分，QPS 约 1 万，
请对比两个方案的优劣并给出推荐"
```

Opus 给出详细的方案对比，包括性能、成本、团队熟悉度等。

## 🧭 深度使用指南

### 如何让 AI 的输出质量更高？

**核心技巧：给 AI 足够的上下文**

| 技巧 | 效果 |
|------|------|
| 给 AI 看代码，不要只描述 | 粘贴关键代码片段，AI 理解更准确 |
| 说明约束条件 | "不能用法 XXX"、"必须兼容旧版本" |
| 给出期望的输出格式 | "用表格输出"、"分步骤说明" |
| 先让 AI 分析，再让它执行 | 先问"你觉得这个需求该怎么设计？" |
| 分步骤，不要一步到位 | 先做分析 → 确认 → 再做实现 |

### AI 协作的黄金法则

```
1. 不要把大需求一口气丢给 AI
   → 分步骤：先分析 → 确认方向 → 再实现

2. AI 生成的代码不等于正确代码
   → 必须审核：逻辑、边界、安全

3. 给 AI 明确的验收标准
   → "这个接口的响应时间必须 < 100ms"

4. 上下文要隔离
   → 后端问题用后端 AI，前端问题用前端 AI
```

### 遇到 AI "失智"怎么办？

**症状**：AI 开始胡说八道、上下文混乱、重复之前的内容

**解决**：
1. 开启新会话（clean session）
2. 重新描述需求，给更少但更精准的上下文
3. 把之前的关键产出粘贴进去
4. 换另一个 AI 工具试试

### 如何用 AI 帮助自己成长？

**记录**：每次 AI 给的建议，记到笔记本里
**复盘**：每周看一次 AI 的输出，总结自己没想到的点
**学习**：让 AI 解释它为什么这么做，学会后就不需要再问
