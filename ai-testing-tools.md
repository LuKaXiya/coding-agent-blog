# 🧪 AI 测试：不是让你少写代码，是让你学会设计测试

> 一个测试工程师的真实困惑：AI 帮我写了测试，但测得对不对我还是不知道。

---

## 一个被忽视的问题：AI 能帮你写代码，但不能替你思考

很多测试工程师用 AI 生成测试用例，然后直接用。

**结果呢？**

要么覆盖率很高但都是无效用例，要么边界情况全漏了 AI 也不知道。

**根本原因是**：AI 能帮你写代码，但它不知道你的业务逻辑，不知道这个功能为什么要这样设计，不知道这个字段的取值范围是多少。

所以用 AI 做测试，有一个前提——**你自己必须知道测什么，才能让 AI 帮你写出怎么测。**

---

## 测试的本质：你想验证什么？

写测试之前，先问自己：

**这个功能的正确性是由什么来定义的？**

一个功能通常有三种正确性定义：

### 1. 业务正确性

这个功能按业务规则执行了吗？

```
举例：用户下单后，库存要扣减，订单状态要变成"已支付"

验证点：
- 支付成功 → 库存扣减了吗？
- 支付成功 → 订单状态正确吗？
- 支付失败 → 库存恢复了吗？
- 重复支付 → 被拦截了吗？
```

### 2. 边界正确性

在极端情况下，这个功能还能正常工作吗？

```
举例：用户积分余额为 0 时，还能扣减吗？

验证点：
- 余额 = 0 → 能扣减 0 吗？
- 余额 = 0 → 扣减 1 失败吗？
- 余额 = 1 → 扣减 1 成功吗？
- 余额 = 1 → 扣减 2 失败吗？
```

### 3. 数据正确性

数据进入系统后，流转正确吗？

```
举例：订单数据从创建到完成

验证点：
- 所有字段都正确记录了吗？
- 时间戳准确吗？
- 状态机转换正确吗？
- 操作日志完整吗？
```

## AI 辅助测试的正确姿势

### 第一步：先告诉 AI 你的业务规则

```
❌ 错误：
"帮我生成积分扣减的测试用例"

✅ 正确：
"帮我生成积分扣减的测试用例，注意：
- 余额不足时扣减失败，返回特定错误码
- 扣减时需要校验风控规则（日限额不能超过 5000）
- 积分有效期默认 1 年，过期后不能使用
- 扣减操作必须幂等
- 并发扣减要保证数据一致性"
```

**核心洞察：你给 AI 的上下文越多，AI 生成的质量越高。**

### 第二步：让 AI 帮你识别你没考虑到的场景

这是 AI 最有价值的地方——**它能想到你没想到的边界情况**。

```
你 → AI：
"我已经列出了主要的业务测试点，请帮我检查还有哪些遗漏：
[你的测试点列表]
"
```

AI 通常能发现：
- 并发场景
- 边界值（最大值、最小值、空值）
- 异常处理路径
- 数据一致性场景

### 第三步：明确告诉 AI 你用的 Mock 规范

Mock 配置是 AI 测试质量的关键。

```
❌ AI 自己猜的 Mock：
when(userRepository.findById(any())).thenReturn(user);

✅ 你告诉 AI 的 Mock：
"所有外部服务调用都要 Mock：
- UserRepository、PointService、PaymentGateway 必须用 @Mock
- 默认返回正常的 Mock 数据
- 需要模拟异常的：网络超时（抛 RuntimeException）、服务限流（抛 RateLimitException）
- 不要 Mock 日期相关的类，用真实时间"
```

## 测试用例设计：你真的知道测什么吗？

很多测试工程师写测试，是按照**代码执行路径**来设计的。

**但正确的方式是按照业务场景来设计。**

### 代码路径视角（不够好）

```
测试：OrderService.createOrder()
路径1：items 非空 → 查询商品 → 校验库存 → 计算价格 → 保存
路径2：items 为空 → 抛出异常
路径3：商品不存在 → 抛出异常
```

### 业务场景视角（更好）

```
测试：用户下单享受折扣
场景1：正常下单，库存充足 → 下单成功
场景2：正常下单，库存不足 → 下单失败，提示库存不足
场景3：下单后，用户取消 → 库存恢复
场景4：下单后，超时未支付 → 自动取消，库存恢复
场景5：下单后，商品降价 → 价格不变，以订单创建时为准
场景6：同一用户重复下单 → 幂等处理
场景7：并发下单，库存只剩 1 件 → 只有 1 个成功
```

**关键区别**：业务场景视角告诉你"为什么测"，代码路径视角只告诉你"怎么测"。

## 用 AI 生成测试用例的 Prompt 模板

### 模板一：按业务场景生成

```
请为 [功能名称] 生成测试用例，按以下业务场景：

1. 正常流程：[描述正常情况]
2. 异常流程：[描述各种异常情况]
3. 边界情况：[描述边界值]
4. 并发场景：[描述并发情况]
5. 数据一致性：[描述数据一致性场景]

要求：
- 每个场景列出：测试目的、输入、预期输出
- 标注哪些是核心场景必须覆盖
- 标注哪些是低频但高风险的场景
```

### 模板二：按接口契约生成

```
请为 [接口名称] 生成测试用例，基于以下接口契约：

请求格式：
[请求参数]

响应格式：
[响应参数]

业务规则：
[业务规则]

请生成：
1. 正常请求的测试用例
2. 参数格式错误（空值、格式不对、类型错误）
3. 参数值超范围（过大、过小、超出枚举范围）
4. 业务规则校验（逐一测试每条规则）
5. 并发请求的测试场景
```

### 模板三：评审 AI 生成的测试用例

```
请帮我评审以下测试用例，指出：
1. 有没有遗漏的重要场景
2. 有没有多余的无效用例
3. 测试数据是否合理
4. 断言是否充分

[粘贴 AI 生成的测试用例]
```

## AI 生成测试的最大陷阱

### 陷阱一：Mock 过度，导致测试没有意义

**问题**：AI 把所有依赖都 Mock 掉了，测试只验证了代码逻辑，但无法验证真实的集成。

**症状**：
```
when(userRepository.findById(1L)).thenReturn(user);
when(orderRepository.save(any())).thenReturn(order);

// 测试通过了，但线上还是报错，因为真实环境数据不对
```

**解决**：哪些该 Mock，哪些不该 Mock
```
✅ 应该 Mock：
- 外部服务（支付、短信、邮件）
- 数据库（可以 Mock 也可以用内存数据库）

❌ 不应该 Mock：
- 你正在测试的那个类本身
- 同一个进程内的简单工具类
```

### 陷阱二：测试数据不真实

**问题**：AI 生成的测试数据过于简单，覆盖不到真实场景。

**举例**：
```
❌ AI 生成的测试数据：
手机号：13800138000（但真实手机号格式更复杂）

✅ 你应该告诉 AI 的：
手机号要符合中国大陆格式：13/14/15/166/17[0-8]/19[0-9]开头，11位数字
```

### 陷阱三：只测Happy Path

**问题**：AI 生成的测试大多是正常流程，异常流程覆盖不足。

**解决**：明确告诉 AI "需要重点测试的异常场景"
```
"重点测试异常场景：
1. 参数为空/null
2. 参数超长（数据库字段长度+1）
3. 并发修改同一数据
4. 外部服务超时
5. 数据库连接失败
6. 幂等性（同一请求重复发送）"
```

## 测试人员的核心竞争力

AI 能帮你写代码，但**设计和判断**必须靠你自己。

### 你需要建立的能力

**1. 业务理解能力**
- 这个功能是做什么的
- 它的正确性由什么定义
- 它的异常边界在哪里

**2. 测试策略设计能力**
- 什么该测，什么不值得测
- 什么先测，什么后测
- 什么用自动化，什么用探索性测试

**3. 测试数据设计能力**
- 边界值怎么选
- 异常数据怎么构造
- 测试数据如何保持独立

**4. 结果判断能力**
- AI 生成的测试，断言对不对
- 覆盖率够了，但核心逻辑测到了吗
- 测试通过了，线上还有风险吗

## 什么时候 AI 测试真的有效

| 场景 | AI 能帮多少 | 你需要做什么 |
|------|------------|-------------|
| 大量重复的 CRUD 测试 | 80% | 审核 Mock 配置 |
| 复杂的边界值计算 | 60% | 告诉 AI 边界规则 |
| 并发场景测试 | 40% | 设计并发场景，AI 执行 |
| 探索性测试 | 0% | 必须人工做 |
| 安全性测试 | 10% | 安全专家介入 |
| 性能测试 | 50% | AI 生成脚本，你分析结果 |

**核心结论**：AI 在"执行"上很强，在"判断"上很弱。你的价值在于判断。

## 行动计划

### 第一周：建立测试思维

- [ ] 拿到需求后，先写测试场景（不是代码），再让 AI 生成代码
- [ ] 用"业务场景视角"代替"代码路径视角"
- [ ] 每个功能问自己：这个功能的核心正确性定义是什么？

### 第二周：Prompt 优化

- [ ] 建立团队的测试 Prompt 模板库
- [ ] 整理常见的 Mock 配置规范
- [ ] 学会让 AI 帮你发现遗漏的场景

### 第三周：质量把关

- [ ] 审核 AI 生成的测试：Mock 是否合理、数据是否真实
- [ ] 检查断言是否充分
- [ ] 建立测试评审机制

### 持续

- [ ] 积累业务理解，这是 AI 替代不了的
- [ ] 关注 AI 测试领域的最新发展
- [ ] 定期复盘 AI 测试的漏报和误报
